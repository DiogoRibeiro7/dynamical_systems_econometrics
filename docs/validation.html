Validation guide

This repository aims to be a credible research package rather than a notebook dump. The validation surface therefore includes schema contracts, hand-crafted mathematical tests, synthetic behavioral tests, smoke entry-point tests, and quality gates for code style and typing.

Core quality commands

pytest -q
ruff check .
mypy

What is tested directly

  • long-format and event-frame schema contracts
  • parameter validation for simulations
  • hand-crafted extremal and recurrence sequences
  • grouped transforms and baseline summaries
  • multivariate stress tables
  • experiment orchestration and wrapper entry points
  • plotting save paths and article-figure bundles

Test categories

  • contract tests: expected columns, duplicates, missingness, and invalid types
  • mathematical sanity tests: simple sequences with known answers
  • synthetic behavioral tests: broad qualitative expectations under fixed seeds
  • pipeline tests: CLI and script entry points that write real outputs
  • numerical edge cases: empty inputs, zero variance, invalid thresholds, and missing mappings

What tests do not prove

  • stable asymptotic extreme-value behavior in short empirical samples
  • causal identification
  • that a synthetic benchmark is a realistic macroeconomic model
  • that a result derived from fallback synthetic data is empirical evidence

Interpretation rule

Passing tests means the repository is internally more reliable. It does not mean that every substantive empirical interpretation is justified. Those questions still depend on data quality, sample size, threshold stability, and the supported-claim guardrails.
