Architecture

The repository is organized as a reusable Python package with thin workflow wrappers around it. The intent is to keep research logic in importable modules, keep orchestration explicit, and keep article outputs traceable back to code and configuration.

Core layers

  • src/dynsys_econometrics/ contains reusable package logic for simulation, transforms, extremes, recurrence, econometrics, experiments, plotting, and contracts
  • scripts/ contains thin entry points for smoke runs, synthetic experiments, empirical runs, figure generation, and data fetching
  • configs/ contains workflow settings that define reproducible parameter bundles
  • tests/ contains contracts, edge-case checks, pipeline smoke coverage, and output-behavior tests
  • docs/ and article/ contain public-facing documentation and writing-oriented assets

Stable interfaces

  • normalized long-format data expectations such as date, series_id, value
  • package-level workflows exposed through the CLI and documented scripts
  • named article-facing figure outputs and their paired CSV sources

Less stable orchestration details

  • intermediate result folder structure may evolve as workflows become clearer
  • experimental comparison scripts may change more often than package contracts
  • empirical adapters may expand as provenance coverage improves

Design rule

If logic is important enough to trust, test, or cite, it should live in the package layer rather than only inside a notebook or an ad hoc script. Scripts should mostly compose package functionality and write named artifacts.
