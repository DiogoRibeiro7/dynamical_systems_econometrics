Data provenance

The repository supports both configured public-data workflows and fully synthetic fallback workflows. Those paths are operationally useful for different reasons, but they must not be merged conceptually: one is about evidence, the other is about reproducibility and method behavior.

Input families

  • public or external macro-financial series fetched or prepared through configured loaders
  • bundled examples and catalogs that describe expected source structure
  • synthetic simulations used for smoke tests, benchmarks, and controlled comparisons

Normalization rule

Regardless of source, loaders are expected to normalize data into a stable long format such as date, series_id, value. That keeps downstream diagnostics source-agnostic and easier to test.

What provenance supports

  • tracing an artifact back to its source path, loader, and configuration
  • distinguishing empirical runs from synthetic fallbacks
  • checking whether a reported output should be treated as evidence or only as a workflow artifact

What provenance does not solve by itself

  • sample-quality problems in the original source
  • threshold instability or estimator sensitivity
  • over-interpretation of visually compelling but weakly supported results

Hard interpretation boundary

Synthetic fallback outputs are legitimate for testing and benchmarking. They are not empirical evidence. If a result matters for a substantive claim, the provenance chain must point to the configured real-data path rather than to a convenience simulation.
