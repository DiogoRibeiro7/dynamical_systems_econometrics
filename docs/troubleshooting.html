Troubleshooting

This page collects the most common operational and interpretation problems in the repository, together with the expected fix or next diagnostic step.

Missing empirical input file

Symptom: the empirical experiment or a notebook cannot find the configured CSV path.

  • Check empirical.input_path in the YAML config.
  • Confirm the file is a local canonical long-format CSV.
  • Use the synthetic workflow first if you only want a smoke test.

Catalog validation fails

Symptom: validate-catalog reports missing fields or an invalid root structure.

  • Ensure the YAML root is a mapping.
  • Ensure series is a list.
  • Each series entry should at least include series_id, description, and source.

Schema mismatch in local CSV files

Symptom: loader functions reject columns or fail datetime/numeric parsing.

  • Normalize inputs to date, series_id, value before downstream analysis.
  • Inspect decimal separators, missing-value markers, and date formatting.
  • Use the data guide page to confirm the expected canonical schema.

Threshold instability

Symptom: extremal-index or recurrence conclusions change materially under small threshold changes.

  • Run threshold sensitivity explicitly before writing strong conclusions.
  • Treat short empirical samples as especially fragile.
  • Reframe the result as exploratory if stability is weak.

No exceedances or too few return times

Symptom: return-time tables are empty or event counts are too small to interpret.

  • Check whether the threshold is too high for the available sample.
  • Increase sample length in synthetic work if the goal is method validation.
  • Do not force interpretation from near-empty event tables.

Optional dependency problems

Symptom: a loader or workflow complains about a package such as yfinance.

  • The base repository should still work without optional online-source dependencies.
  • Install the missing package only if that specific source is needed.
  • Prefer local CSV inputs when reproducibility matters more than live fetching.

Figure/script outputs differ from notebook outputs

Symptom: a notebook and a script produce different-looking outputs.

  • Treat script-generated outputs as the publication path.
  • Use notebooks as exploratory and explanatory layers.
  • Check whether a notebook is using fallback synthetic data while the script is using a local raw file.

Interpretation problem rather than code problem

Symptom: the code runs, but the claim you want to write sounds too strong.

  • Check the research guardrails page.
  • Check whether the claim is descriptive or causal.
  • Check whether the result comes from real empirical data or a synthetic fallback.
