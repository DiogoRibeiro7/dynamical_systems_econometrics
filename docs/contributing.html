Contributing guide

Contributions are useful when they improve the package, the public documentation, or the research workflow without making the evidence story less disciplined. This is a research repository, so correctness and interpretability matter more than feature count.

Good contribution targets

  • bug fixes in estimators, loaders, plotting, or CLI wrappers
  • tests that tighten contracts or cover numerical edge cases
  • documentation improvements that clarify supported workflows
  • configuration or automation improvements that increase reproducibility

Changes that need extra care

  • new empirical claims based on weak or fallback data
  • method changes that alter estimator meaning without clear documentation
  • silent output-path changes that break article or workflow references
  • new notebooks that duplicate package logic instead of reusing the library

Suggested contribution flow

git checkout -b topic/short-description
pip install -e ".[dev]"
pytest -q
ruff check .
mypy

Review standard

A strong contribution states what changed, why the change is correct, what outputs or docs are affected, and whether any research interpretation boundaries moved. If the answer to that last question is unclear, the change is not ready.

Research-specific rule

Do not turn convenience outputs into evidence claims. Synthetic smoke runs, fallback datasets, and visually suggestive plots are useful, but they are not automatic support for a macro-financial conclusion.
