Citation

This repository should be cited in a way that preserves the difference between software, workflow artifacts, and substantive research claims. A citation is strongest when it points to a stable repository state and makes clear whether the referenced object is code, documentation, or an article-facing output bundle.

Preferred repository citation basis

  • use the metadata in CITATION.cff as the primary reference source
  • prefer tagged releases over moving branch heads when a stable version exists
  • name the repository directly when discussing software, package structure, or reproducible workflows

What to cite as software

  • package code under src/dynsys_econometrics/
  • documented CLI and script workflows
  • versioned repository states that produce stable public artifacts

What to cite more carefully

  • article-facing figures should be cited as outputs derived from the repository, not as standalone evidence engines
  • synthetic benchmark outputs should not be cited as empirical findings
  • draft article material should be distinguished from stable software interfaces

Attribution rule

If a downstream user relies on the code, cite the repository. If they rely on an argument built from repository outputs, they should cite the argument-bearing material as well and keep the software citation separate from the evidence citation.
