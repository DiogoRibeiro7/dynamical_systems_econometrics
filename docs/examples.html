Empirical input examples

The repository now supports several empirical input styles. This page shows the intended examples and when each one is the right choice.

1. Multi-series local config

Use the default empirical config when you already have a few local files and want explicit per-series transformations in one place. 

python -m dynsys_econometrics empirical --config configs/empirical.yaml

2. Single catalog

Use a catalog when source metadata and transformations should travel together. The repository example catalog is still the simplest metadata-first path. 

python -m dynsys_econometrics materialize-catalog --catalog data/catalog.example.yaml
python -m dynsys_econometrics empirical --config configs/empirical.yaml

3. Multi-catalog config

Use multiple catalogs when source families are easier to maintain separately but should still be analyzed together in one run. 

python -m dynsys_econometrics empirical --config configs/empirical_multicatalog.yaml

4. Cached FRED config

Use a cached FRED config when you want an offline-first empirical path that can refresh a canonical cache deliberately instead of silently depending on network access every run. 

python -m dynsys_econometrics empirical --config configs/empirical_fred_cached.yaml

5. Cached yfinance config

Use a cached market-data config when you want the same offline-first discipline for price proxies or ETF-style stress indicators. 

python -m dynsys_econometrics empirical --config configs/empirical_yfinance_cached.yaml

Interpretation rule

  • local examples are for structure and reproducibility
  • cached remote configs are for controlled refresh workflows
  • synthetic or fallback paths are not empirical evidence
